## 二、简历片段（可直接粘贴）

### 项目名称

**Steam 创意工坊多阶段数据采集与分析流水线**（个人项目）  
2025.05 — 至今 · Python · 独立设计与实现

---

### 一行摘要（放项目标题下）

输入单个 Steam APPID，自动完成工坊模组从**列表发现 → 详情抓取 → 解析建库 → 统计分析**的全链路；支持 TB 级 HTML 增量续跑与三阶段断点恢复，已覆盖群星/V3/钢丝等 P 社游戏及 10 款目标游戏规划（合计约 58 万模组规模）。

---

### 项目描述（2–3 句，投递数据开发 / 采集岗时用）

独立从零搭建面向 Steam 创意工坊的数据工程流水线，将分散在三个子系统中的采集、爬取、分析模块统一为可一键运行的 CLI 总控。针对 Steam 平台「每标签最多约 5 万条」「1667 页硬上限」等约束，设计 tag 交叉 + 无标签全站列表的覆盖策略；针对详情页单 IP 限流，实现 asyncio 多端口爬虫与 SQLite 任务状态机，崩溃后可无损续跑。项目约 **58 个 Python 模块、1.2 万行代码**，含冒烟预检、结构化错误码与数据防误删机制。

---

### 核心职责与亮点（选 4–6 条放简历，与普通课设/CRUD 实习社区分）

**① 三阶段 ETL 总控与断点恢复（数据工程向）**  
设计 `simple_info → detail_fetch → analysis` 流水线及 `pipeline_state.json` 全局状态；子包内另有 `mod_fetch.sqlite`、HTML 目录级细粒度续跑。无状态文件且数据目录非空时拒绝启动，避免误覆盖已爬取的 GB 级数据；失败时输出带错误码、原因与修复建议的 `PipelineError`（6 类业务错误）。

**② 平台约束下的采集覆盖率方案（领域 + 工程）**  
调研并落地 Steam 浏览页分页上限（1667 页 × 30 条/页）：按官方 tag 展开 URL + 合成 `No_selected_tag` 全站列表，在平台规则内最大化 mod 发现率；浏览页缺口检测脚本 + 单轮最多 5 次补爬，输出 `browse_html_gaps.json` 可追溯缺口。

**③ 生产向异步爬虫（可独立复用的通用组件）**  
自研 `AsyncCrawler`：aiohttp 并发、多 egress 端口池、healthy → degraded → quarantined 熔断、HTTP 429 + Retry-After 解析、传输层指数退避；爬虫与业务解耦，输入 URL 列表 + SQLite 状态即可爬任意 HTML 站点。清除系统代理环境变量，仅走配置端口（适配 Steam++ 等本地 MITM 场景）。

**④ HTML → 关系型数仓 + 自动化分析报告**  
详情页 HTML 解析为多表 SQLite（mods、tag 排名、browse 元数据、曝光比等），一键导出 Excel；分析层含订阅量 Gini 系数、Dagum 组间分解、按订阅分层曝光统计及 matplotlib 图表，支持多游戏 profile 切换输出。

**⑤ 可运维性：跑前冒烟、跑中可观测、跑后可验收**  
`test/smoke.py` 预检配置可读、数据目录可写、APPID 是否含工坊、代理端口连通、aiohttp 依赖就绪；总控支持 `--status` / `--only` / `--from` 分阶段执行与 dry-run；统一路径布局 `lib/paradox_paths.py`，支持数据目录外置到绝对路径。

**⑥ 规模与交付物（用数字说话，区别于「调个 API 的小脚本」）**  
单游戏详情页量级：钢丝约 6.3 万、天际线约 36.9 万 URL；浏览页阶段天际线约 1.3 万页、群星约 2000 页。交付物含原始 HTML、多层 SQLite、Excel 总表、Markdown/图表分析报告——完整数据工程闭环，而非仅展示型 Demo。
